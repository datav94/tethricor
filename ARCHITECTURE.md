# System Architecture
## Universal Harness Abstraction Layer

### 1. High-Level Components
* **Client App:** The developer's microservice or application. Sends instructions via REST or WebSockets to `localhost`.
* **Harness Sidecar:** An injected container running the hardened open-source harness (e.g., Pi, OpenHands).
* **Mutating Webhook (Kubernetes control plane):** Intercepts pod deployments, reads `harness.yaml`, and injects the Harness Sidecar. Pure Kubernetes `AdmissionReview` — runs on AKS, GKE, EKS, kind, minikube, or any other cluster.
* **LLM Gateway:** The centralized, OpenAI-compatible LLM gateway (your own, or a self-hosted proxy like LiteLLM/Portkey/Kong AI Gateway) — resolved via `model.routing_profile`, never hardcoded to one vendor.
* **MCP Server:** The server providing approved tools/skills to the sidecars, reached through the same gateway.
* **Sandbox Runtime:** The secure execution environment where the harness actually runs code — a pluggable `SandboxProvider`. No default backend is assumed; point it at a service you self-host or operate, or use an opt-in isolation package (`microsandbox`, `e2b`).

### 2. Request Flow
1.  **Deployment:** Developer pushes code with `harness.yaml`. The Kubernetes Mutating Webhook injects the harness sidecar and mounts environment variables (identity federation, e.g. Azure Workload Identity, is opt-in and cluster-specific — see `tethricor.internal/identity-mode`).
2.  **Task Initiation:** The Client App sends a task `{"task": "Analyze data"}` to the Harness Sidecar over standard local REST/WS.
3.  **Agent Loop:** The Harness Sidecar queries the **LLM Gateway** to process thoughts via the LLM.
4.  **Tool Execution:** When the harness needs to use a tool, it fetches skills from the **MCP Server**.
5.  **Code Execution (Sandboxed):** If the harness tries to run arbitrary code (e.g., Python, Bash), the hardened container intercepts the call and forwards it via WS/REST to the configured **Sandbox Runtime** provider.

### 3. Local Development Parity
Developers use a bundled `docker-compose.yaml` generated by the CLI, which emulates the Kubernetes sidecar pattern locally (against local doubles for the gateway, MCP server, and sandbox runtime), ensuring parity before cloud deployment.

### 4. Known gap: harness toolchain provisioning
The sandbox session a harness's forwarded command runs in must already have that
harness's own runtime/toolchain installed (not just a generic language image) for
execution to actually succeed. Tethricor does not currently provision per-harness
sandbox images automatically — see `docs/FRAMEWORK_GUIDE.md` §13 for detail and status.
